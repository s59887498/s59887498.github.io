QA Engineer Onboarding

前三個月職能培養圖表

這份圖表把新人培養拆成三個月：第 1 個月先會照既有案例穩定執行， 第 2 個月開始參與 EOMC Test Planning 與 Test Case 產出，第 3 個月朝能從需求到 UAT 驗收獨立推進的 QA Owner 前進。

01 第 1 個月

建立基礎

建立 QA 基礎與產品熟悉

目標不是獨立負責，而是學會 QA 的基本工作方式：能照案例執行、留下正確證據、遇到不清楚的地方會標註。

定位：照做與記錄

他要會

  • 看懂 PRD / UI mockup 的基本結構。
  • 知道 Test Case 欄位與執行方式。
  • 會截圖、錄影、記錄實際結果。
  • 會寫基本 Bug Ticket。
  • 知道 Happy Path：正常流程。
  • 知道 Exception Handling：例外處理。
  • 知道 Boundary / Edge Case：邊界條件。
  • 知道 Permission / State / Integration：權限、狀態、串接。
  • 知道 UI / UX / Content：畫面、文案、Mockup 對版。
  • 知道測試環境、帳號、測試資料不能亂用。

適合任務

  • 跟著資深 QA 跑 Smoke Test。
  • 跟著資深 QA 跑回歸測試。
  • 跟著跑權限驗證。
  • 幫忙整理測試證據。
  • 把口頭測試步驟整理成 Markdown / Excel。

DoD

  • 可以照測試案例完整執行。
  • Bug Ticket 不缺重現步驟、實際結果、預期結果、截圖。
  • 不清楚的地方會標註，不會自行補需求。
  • 星舟快充專案：完成上線前回歸測試（Stage 環境）。
  • EOMC 專案（企業營運管理中心）：完成 S11 決策中心 L1 / L2 集團財務、管理系統：集團財務測試執行與證據整理。
本月重點

先建立 QA 工作紀律：照案例執行、正確記錄、不要把推測當需求。

能力進展
照做 測試設計 獨立 Owner
02 第 2 個月

開始設計

開始寫簡單 Test Case

目標是從「照做」進到「可以參與測試設計」，能依 PRD / RA / Figma 產出可執行、可 review 的 Test Case 初稿，並開始理解專案追蹤節奏。

定位：測試設計參與

他要會

  • 依照 PRD / 畫面補 Happy Path 正常流程。
  • 補基本 Exception Handling，例如必填未填、格式錯誤、無權限。
  • 補基本 UI copy / layout 檢查。
  • 協助整理 Test Report 的測試結果。
  • 知道 Passed / Failed / Skipped 的判斷方式。

適合任務

  • EOMC Test Planning & 產出 Test Case（S14 資產管理、S15 相關功能）。
  • 問題基本追蹤 & 專案進度持續進行（Asana Ticket）。
  • 測試結合 AI 應用分享。

DoD

  • 可以寫出可執行的 Test Case。
  • Steps 清楚，Expected Result 能對應步驟。
  • 測試結果能被 QA / RD 讀懂。
  • 需要資深 QA review，但不是從零重寫。
本月重點

讓新人開始把需求轉成 Test Plan / Test Case，並透過 Asana 建立問題追蹤與專案進度感；同時練習整理 AI 如何輔助測試設計與日常 QA 工作。

能力進展
照做 測試設計 獨立 Owner
03 第 3 個月

嘗試負責

朝獨立 QA Owner 前進

目標是讓新人能從專案需求理解、測試設計、進入測試到 UAT 驗收持續推進，不只停留在單一小範圍任務；過程仍有資深 QA 協助 review、補位與風險判斷。

定位：獨立作業 Owner

他要會

  • 從 PRD / RA / Figma 理解專案需求與測試範圍。
  • 規劃 Test Plan、設計 Test Case，並能說明 coverage 與測試風險。
  • 進入測試後能安排執行節奏、追蹤 Bug、同步 Asana / 專案進度。
  • 整理測試結果、未解風險與 UAT 驗收重點。
  • 知道哪些問題可以自己判斷，哪些需要升級給資深 QA / PM / RD。
  • 開始把 API / RBAC / 資料驗證納入測試設計，而不是只看 UI 流程。

適合任務

  • 可獨立負責一個專案功能的 QA 流程，從需求確認到 UAT 驗收。
  • 跨 PM / RD / QA 協作的功能測試與問題追蹤。
  • 需要整合 Test Plan、Test Case、Bug、風險與驗收結果的專案。

DoD

  • 可以獨立拆解需求、規劃測試範圍並產出 Test Plan / Test Case。
  • 可以推進測試執行、追蹤問題、整理測試結果與未解風險。
  • 可以支援 UAT 驗收，清楚說明已測範圍、未測範圍與 release 風險。
  • 重大風險、放行判斷與跨部門爭議仍會找資深 QA 一起確認。
本月重點

培養更完整的 QA ownership：能自己接住需求、設計測試、推動執行與支援 UAT，但在關鍵風險與放行判斷上仍有資深 QA 輔助。

能力進展
照做 測試設計 獨立 Owner
