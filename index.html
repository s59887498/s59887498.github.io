What Is Built

已建成的是 QA Agent OS 的底層

從 Prompt 走向外部化的 QA 知識、方法、能力與交付規則，讓品質工作可以被重複執行與持續演進。

Core Value

不是讓 AI 自由發揮，而是讓 AI 在 QA 品質系統中工作。

需求進來後，系統能辨識來源、選擇方法、載入對應能力、檢查前置條件，並依交付契約決定是否可以輸出或必須停下來。

核心目標：把 QA 判斷依據變成可追溯、可測試、可治理的 operating layer。
01 Knowledge

QA Domain Package

把需求來源、Theory、Methodology、Examples、Templates 與專案規則分層，避免知識只藏在單一 Prompt。

Knowledge Contract：每一層有自己的責任與 dependency direction。
02 Skills + Methodology

能力與做法都被契約化

Skill 不是泛用指令；它依 Test Design Map、測試技法選擇、輸入 Gate 與 Output Contract 執行特定 QA 任務。

Skill Contract：可發現、可路由、可驗證，也不重複維護共用知識。
03 Runtime

Workflow 有明確的控制面

五種 Mode、PASS / PARTIAL / BLOCK、repo routing 與安全停止規則，讓不同交付物有正確的前置條件與責任邊界。

Runtime Contract：只做可驗證的部分，不以猜測補足缺漏需求。
Evaluation Foundation

品質閉環的第一層已存在

Test Case 已具 deterministic validator、framework validator、固定 fixture 與 rubric；它們是未來 score、judge、feedback 與 retry loop 的可信起點。

下一步不只是多一個 Agent，而是把品質標準轉成可重複執行的 Evaluation Framework。

Evaluation Framework

讓 QA Agent 對品質負責的方式

Eval 不是叫 AI 自己說「我做得好不好」，而是先定義標準，再以 deterministic checks、golden fixtures 與 calibrated judge 驗證產出。

1

Generate

依 Knowledge、Methodology 與 Skill Contract 產生 Test Plan、Case、automation handoff 或 Bug Draft。

2

Contract Eval

先檢查欄位、格式、ID、steps、traceability、source grounding 與 output path 等 deterministic 規則。

3

Quality Rubric

以 coverage、boundary、negative、business rule、risk、duplicate、readability 等 QA 指標評估內容品質。

4

Judge Calibration

用 golden fixtures 與明確 rubric 校準 LLM judge；judge 的工作是依標準評分，不是自由改寫需求。

5

Score and Feedback

分數、缺口與建議被結構化輸出，作為人工 review 或下一次生成的具體品質訊號。

6

Bounded Retry

只有明確可修復的項目才有限次重試；規格矛盾、缺資料與高風險決策一律交回 owner。

Next Build

Evaluation-first Roadmap

先讓品質標準可量化，再讓 Agent 根據可信的訊號採取下一步。

P1Quality Model

定義跨產物評估 taxonomy

為 Test Plan、Test Case、automation handoff、Bug Draft 建立可追溯的品質維度、pass threshold 與 failure code。

產物：Eval schema、rubric、scorecard、failure taxonomy。
P2Golden Data

擴充 fixtures 與 deterministic checks

建立正常、邊界、缺規格與反例資料集，將欄位契約、coverage 與 traceability 檢查自動化。

產物：golden dataset、validator regression、expected score。
P3LLM Judge

校準內容品質評估器

由 rubric 驅動 Judge，回傳各維度分數、證據與缺口；與 human review 樣本比對，避免漂移。

產物：judge prompt contract、calibration set、review agreement。
P4Controlled Loop

把評估結果接成受控修正

只有 score 未達標且 failure 可修復時才重跑，保留 run manifest、retry budget、diff 與人為核准點。

產物：eval orchestrator、feedback packet、bounded retry policy。
P5Memory + CI

建立品質歷史與持續回歸

保存 run state、score trend、失敗類型與 review decision，並在變更 Skill 或 Knowledge 時自動跑 Eval regression。

產物：run history、quality dashboard、CI evaluation gate。

Operating Boundaries

Evaluation-driven 系統仍必須受治理

品質閉環的目的不是讓 AI 自行擴權，而是把判斷、證據與責任切得更清楚。

來源不足時停止

Evaluator 不能替需求補規格；缺少來源時輸出 gap 與確認項目。

評估與生成分工

Judge 依 rubric 與 evidence 評分；不得以自我感覺取代可重複的標準。

重試次數有上限

每次 retry 必須有 failure code、修正範圍與 retry budget，避免無限 loop。

人仍保有決策權

正式 release、風險接受、需求衝突與 destructive action 必須由 owner 核准。
