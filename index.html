What Is Built

目前已建成的 QA Agent

它不只是把文字交給 AI，而是把品質工作中需要判斷、追溯與停止的規則固定下來。

Core Value

從 Prompt 驅動，走向可治理的 QA 工作流。

輸入需求與範圍後，Agent 能判斷該走哪個 Mode、缺什麼、可以做什麼，以及什麼情況必須停止。

核心目標：降低人工作業成本，同時保留品質判斷與交付證據。
01 Workflow Control

五種 Mode 與 Gate

Test Plan、Test Case、Automation、Test Report、Full E2E 依任務選擇，不用同一份 checklist 卡住所有工作。

PASS / PARTIAL / BLOCK、Readiness Checklist、Output Contract。
02 QA Knowledge

測試思維已被沉澱

把來源優先順序、Test Design Map、測試技法、範本、專案規則與品質閘分層維護。

AGENTS、Methodology、Theory、Examples、Templates、Skills。
03 Cross-repo Delivery

文件與 automation 正確分流

QA repo 保存來源、Test Plan、Case、Report；正式 API、UI、OCPP automation 放入 Automation repo 既有框架。

保留可追溯 mapping，避免文件與可執行程式混放。
Foundation

Evidence、Safety、Traceability

Test Case 有 deterministic validator；正式報告要求 execution evidence；敏感資訊以環境變數管理；destructive test 必須取得當次允許。

Agent 可以前進，也知道何時不該自行前進。

Next Architecture

從 Workflow 到 Loop Agent

下一步不是先增加更多 AI 角色，而是讓現有 Agent 能在安全邊界內形成可控的執行閉環。

1

Run State

建立每次執行的 Run Manifest：來源版本、scope、選取案例、環境、允許操作與目前狀態。

2

Execute & Collect

執行已核准的 automation，收集 test result、log、trace、screenshot 與 cleanup 狀態。

3

Classify Failure

區分產品缺陷、測試程式、環境、測資、權限與規格缺口，避免把所有 fail 都拿去重跑。

4

Repair Within Policy

只修正被允許的測試設定、selector、等待條件或測資準備；產品行為問題不得自行掩蓋。

5

Retry With Budget

限定重跑次數與條件。重試後仍失敗、缺權限或來源不明時，明確結束為 FAIL 或 BLOCK。

6

Evidence & Escalate

保存結果與下一步，產出 Report 或 Bug Draft；需要人做決策時再交由 QA / RD / PM review。

Execution Roadmap

下一階段要完成什麼

先把單一安全 loop 跑通，再接 CI 與治理；每一階段都有明確的完成條件與停止邊界。

P1

建立 Run Manifest 與狀態模型

記錄任務輸入、來源版本、test mapping、環境、狀態、evidence 與 cleanup。

完成條件：任一 automation run 可中斷、續跑、回查與完整追溯。

P2

建立安全的 execute-classify-retry loop

先從 staging 或 QA 環境的 API regression 開始，導入失敗分類、可修範圍與 retry budget。

完成條件：每一個 FAIL 都有分類、證據與停止或下一步決策。

P3

導入 review 與品質變更守門

把 Ready-for-Automation、selector contract、test data、review result 與 code mapping 形成可審查鏈。

完成條件：AI 產物與 automation 都能被人接手、review 與維護。

P4

串接 CI / Jenkins 與治理指標

由 PR、Nightly、Release 或 synthetic event 觸發可信 automation，回收 trend、report 與 issue draft。

完成條件：品質結果不靠人工追問，且能用 evidence 支援放行與風險決策。

Safety by Design

Loop Agent 的不可跨越邊界

自主不代表無限制。真正可信的 loop，必須清楚知道哪些事可以做、哪些事要停下來交給人。

產品行為與需求衝突

不自行修改 Expected Result 或補造規格；保留差異與影響，交由 owner 決策。

Destructive 操作

production 寫入、退款、刪除、批次異動預設不執行；每次任務都需要明確允許。

Retry Budget

重跑有明確次數與條件；避免環境或產品問題被無限重試掩蓋。

Human Approval

Bug 正式開單、release decision、risk acceptance 與重大測試變更仍由人負責。
